# Observabilidade

- Rule ID: R-O11Y-001
- Severidade: guideline com pontos hard para visibilidade de falhas
- Escopo: runtime, providers, persistência e CLI.

## Objetivo
Garantir visibilidade operacional suficiente para diagnosticar falhas de execução sem poluir a experiência do usuário.

## Requisitos

### Logging
- Logging estruturado deve usar `log/slog` como padrão do projeto.
- O logger deve ser injetado por construtor ou composition root; evitar dependência implícita em logger global.
- `Logger.With` deve ser usado para carregar contexto estável de `run_id`, `workflow`, `step` e `provider`.
- `WithGroup` deve ser usado quando campos relacionados precisarem de namespace consistente.
- Em caminhos quentes ou repetitivos, preferir `LogAttrs` e `slog.Attr` para reduzir alocação e evitar pares chave-valor frágeis.
- Campos mínimos quando aplicável: `run_id`, `workflow`, `step`, `provider`, `operation`, `duration_ms`.
- Não logar segredos, tokens, prompts sensíveis ou conteúdo inteiro de artefato sem necessidade.
- Quando um tipo carregar dado sensível, preferir `LogValue` para redigir ou agrupar a representação no log.

### Tracing e Tempo
- Operações longas devem capturar duração.
- Execução de subprocesso, persistência e parsing devem ser instrumentados pelo menos com timing e erro.
- O runtime deve medir latência entre término do provider e exibição do prompt HITL.
- Quando contexto estiver disponível, usar métodos de logging que recebam `context.Context` para permitir correlação por handler.

### Auditoria
- Operações de filesystem executadas por steps devem ser registradas para auditoria.
- Criação, edição, retry, pausa e retomada de run devem gerar trilha mínima de eventos.

## Proibido
- `fmt.Println` como mecanismo principal de logging operacional.
- Misturar output de UX do terminal com logs de aplicação no mesmo fluxo sem propósito claro.
- Logar credenciais ou payload sensível integral.
- Omitir contexto básico quando um step falhar.
